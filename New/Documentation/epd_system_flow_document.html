The system transforms quarterly Excel workbook data into a persistent, queryable incentive ledger. Four pipeline stages move data from source to dashboard.

B. Module Breakdown

M1 Staff Management

Maintains the roster of Medical Representatives. Each rep is linked to quarterly activity records that determine reimbursable eligibility.

Core tablesrepresentative, rep_quarter_status
Key logicIf months_worked = 3 → reimbursable_months_pct = 100%. Partial months reduce payout proportionally.
InputStaff Input sheet rows; "Worked" / leave status per quarter column

M2 Product / Portfolio Mapping

Defines the promo-line structure. Each promo-line contains up to 4 portfolios; each portfolio groups one or more products and carries a share weight that determines its contribution to the total incentive.

Core tablespromo_line, portfolio, product
ConstraintPortfolio share weights within a promo-line must sum to 100%
Variable productsProducts are named per product.name; UI resolves "Product 1" dynamically from portfolio.portfolio_index + product.name

M3 Sales vs Plan Calculation Engine

Compares actual sales against plan for each rep × product × quarter combination and produces a percentage achievement score per portfolio.

Core tablessales_plan, sales_actual
Outputachievement_pct per portfolio, stored in portfolio_incentive_split

M4 Incentive Calculation Engine

Core business logic module. Applies payment coefficients to achievement scores, computes sales result incentive and TCFA incentive, and aggregates the final payout per rep per quarter.

Core tablespayment_coefficient, call_actual, incentive_result, portfolio_incentive_split
TriggerRuns after all ACT and PLAN data for a quarter is loaded and verified

M5 Reporting Layer

Exposes aggregated data to the dashboard and external exports. Supports per-rep, per-quarter, and cross-quarter views with LC/USD toggle.

Viewsv_rep_summary, v_portfolio_breakdown, v_quarter_leaderboard
ExportCSV and XLSX download endpoints; data locked in LC at time of finalization

C. Incentive Calculation Logic

Step 1 — ACT vs PLAN (Sales Achievement %)

For each rep, each portfolio is scored by comparing total actual sales to total planned sales across all products in that portfolio.

Portfolio_Achievement% = (Σ ACT_value_LC across products in portfolio) ÷ (Σ PLAN_value_LC across products in portfolio) × 100 Example: Portfolio 1 has 2 products. ACT = 85,000 LC, PLAN = 100,000 LC → Achievement = 85.0%

Step 2 — Payment Coefficient Lookup

Achievement % is mapped to a multiplier via a banded lookup table (payment_coefficient). This non-linear mapping rewards over-achievement and penalizes under-achievement.

Achievement rangeMultiplierTier label
0% – 79%0.00Below threshold
80% – 89%0.50Partial
90% – 99%0.80Near target
100% – 109%1.00On target
110% – 119%1.20Over target
≥ 120%1.50Exceptional
Assumption: Exact coefficient bands are not specified in source document. The table above is an illustrative but realistic approximation. The actual bands must be sourced from the Excel Assumptions Calc and seeded into the payment_coefficient table before go-live.

Step 3 — TCFA (Target Call Frequency Achievement)

TCFA measures whether the rep made the required number of customer calls during the quarter.

TCFA% = (Actual_Calls ÷ Planned_Calls) × 100 Example: 87 actual calls ÷ 100 planned = 87.0% TCFA

TCFA is treated as a separate incentive component with its own weight. A TCFA coefficient (from the same payment bands or a dedicated TCFA band table) converts TCFA% into a TCFA multiplier.

Step 4 — Sales Result Incentive

Calculated portfolio-by-portfolio, then summed. Each portfolio's contribution is weighted by its share_weight_pct.

Portfolio_Contribution_LC = Target_Incentive_Base_LC × Portfolio_Share_Weight × Payment_Multiplier Sales_Result_Incentive_LC = Σ Portfolio_Contribution_LC (across all portfolios) Example: Base = 500,000 LC. Portfolio 1 weight = 40%, multiplier = 1.00 → contribution = 200,000 LC. Portfolio 2 weight = 30%, multiplier = 0.80 → contribution = 120,000 LC. Portfolio 3 weight = 30%, multiplier = 1.20 → contribution = 180,000 LC. → Sales Result Incentive = 500,000 LC

Step 5 — TCFA Incentive

TCFA_Incentive_LC = Target_Incentive_Base_LC × TCFA_Weight_Pct × TCFA_Multiplier Assumption: TCFA carries a fixed weight (e.g. 20%) of the base. Exact weight must be confirmed from source data.

Step 6 — Final Incentive Derivation

Adjusted_Base = Target_Incentive_Base_LC × Reimbursable_Months_Pct Total_Incentive_LC = (Sales_Result_Incentive_LC + TCFA_Incentive_LC) × Reimbursable_Months_Pct Total_Incentive_USD = Total_Incentive_LC ÷ Quarter.Exchange_Rate_LC_USD Reimbursable_Months_Pct = months_worked / 3. A rep on leave for 1 of 3 months → 66.67%

D. Data Modeling Decisions

DecisionRationale
payment_coefficient as a tableBands change between incentive periods. Storing them as rows (with an optional effective_from date) allows history without code changes.
portfolio_incentive_split separate from incentive_resultThe result entity holds one row per rep/quarter. Splits hold one row per portfolio per result. This avoids wide, sparse columns for variable portfolio counts.
Variable products resolved via portfolio_indexPromo-lines have different product sets. UI labels ("Product 1") are derived dynamically from portfolio.portfolio_index + product.name rather than hard-coded columns.
Exchange rate on quarter, not incentive_resultAll reps in a quarter use the same rate. Storing on the quarter row prevents duplication. Historical data is locked in LC; USD is always a derived display value.
call_plan and call_actual separate from sales tablesTCFA is an independent KPI with its own calculation path. Mixing with sales rows would require NULLs or sparse columns.
rep_quarter_status.reimbursable_months_pct pre-computedComputed once at ETL time from months_worked. Avoids re-derivation in every incentive query.

Upload and import strategy

The recommended approach is a standardized import template rather than accepting the raw Excel layout. The ETL parses raw XLSX on the backend using a column position map; the user uploads the same file format they already produce. A validation report is returned before data is committed, surfacing missing columns, mismatched reps, and out-of-range values. This directly addresses the open question about unnamed columns in row 8.
